# Design Interrogation（Grill）指南

## 核心理念
动手实现之前，AI 沿决策树逐分支拷问你，直到双方对计划有共识。
把说不出口的假设摊开，让风险提前冒出来。

## 什么时候触发
- 用户提出一个新的功能需求或设计变更
- 用户要求实现一个嵌入式模块（驱动、BSP、协议栈等）
- 用户描述了一个方案但没有给出具体实现细节

## 执行流程

### 1. 先探索已有代码库
如果有现成代码，先读相关模块（现有驱动、配置、硬件抽象层），再提问。
不要在空白上下文里连环拷问。

### 2. 沿决策树一个一个问
一次只问一个问题，等用户回答后再继续：
```
Q1: "这个 PWM 通道的目标频率和占空比范围是多少？
    推荐方案：根据现有 timer 驱动，建议使用 TIM2，
     因为 TIM1 已在电机控制占用。"
Q2: "中断优先级怎么定？高于现有 AdcDrv 还是低于？
     如果高于，注意共享数据区需要关中断保护。"
...
```

### 3. 问题的来源
- 能从代码库推断的 → AI 探索后给出推荐答案并确认
- 用户或 AI 能拍板的 → 用户回答 + AI 推荐方案
- 需要业务/硬件确认的 → 记录待确认项

### 4. 输出
- 更新 `CONTEXT.md` 或架构决策记录（ADR）
- 达成共享理解后再进入代码实现

## 典型的拷问方向（嵌入式场景）
| 方向 | 典型问题 |
|------|---------|
| 硬件约束 | 用什么 MCU/外设？引脚冲突？时钟树？ |
| 时序要求 | 中断频率？DMA 带宽？任务周期？ |
| 资源边界 | RAM/Flash 余量？堆栈大小？ |
| 接口契约 | 高低电平有效？SPI 模式？I2C 地址？ |
| 错误策略 | 超时处理？重试次数？故障恢复路径？ |
| 已有冲突 | 跟现有驱动重叠？中断号冲突？DMA 通道冲突？ |
| 测试验证 | 怎么测？硬件在环？模拟器？ |
