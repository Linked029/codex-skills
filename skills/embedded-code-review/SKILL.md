# 嵌入式 C/C++ 代码审查

## 核心原则
**先看会不会死机、丢数据、破坏硬件状态、影响实时性。风格建议往后放。**


## 审查流程

### 阶段 0：设计拷问（Grill）

**如果你给我的是设计方案、需求描述或尚未实现的代码，先执行本阶段。**

逐分支拷问，一次一个问题，等用户回答后再继续。可以参考 `references/grill-design.md`。

#### 嵌入式场景的拷问方向
- 硬件约束：MCU 型号？外设冲突？引脚分配？时钟树？
- 时序要求：中断频率？DMA 带宽？任务周期？
- 资源边界：RAM/Flash 余量？堆栈大小？
- 接口契约：高低电平？SPI 模式？I2C 地址？
- 错误策略：超时处理？重试次数？故障恢复路径？
- 已有冲突：跟现有驱动重叠？中断/DMA 通道冲突？
- 测试验证：怎么测？硬件在环？

#### 决策树建模
```
硬件约束 → 时序 → 资源 → 接口 → 错误处理 → 冲突检查 → 验证
```
每走完一个分支，确认该分支决策后再进入下一个。

#### 输出
- 记录待确认项（需要业务/硬件确认的）
- 达成共享理解后，再审查具体代码
## 输出格式

逐项输出审查结果，按严重程度标记：

| 标记 | 等级 | 含义 |
|------|------|------|
| 🔴 **严重** | P0 | 正确性缺陷，必须修复 |
| 🟠 **高风险** | P1 | 嵌入式专属风险，强烈建议修复 |
| 🟡 **中风险** | P2 | 实时性问题，建议修复 |
| 🟢 **低风险** | P3 | 风格和规范建议 |

每条审查结果包含：
- 问题类型（对应上述优先级）
- 代码位置（文件名:行号）
- 问题描述
- 风险影响说明
- 修复建议（含代码示例）

## 审查原则
- 理解代码的硬件上下文（芯片型号、外设资源、编译配置）
- 建立"软件逻辑 ↔ 硬件约束"的双向映射
- 对可疑代码实施"双重校验"：静态分析 + 逻辑推理
- 参考 eferences/checklist.md 获取详细检查项说明
- 参考 eferences/examples.md 获取典型问题示例
